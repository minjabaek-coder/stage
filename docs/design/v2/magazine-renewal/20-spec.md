## 1. 데이터 모델 명세

### 1.1 콘텐츠 모델 (변경 최소)
- `Magazine` 1-N `MagazinePage`(`kind` 분기) 유지. `@@unique([magazineId, sortOrder])` + 2-pass `applyPageOrder` 유지.
- **이미지형 `kind=image`**: `imageUrl` 그대로. 렌더/저장 경로 **무변경**.
- **구성형 `kind=composed`**: `layout`(Json?) 컬럼을 **v2 스키마로 진화**(아래). **DB 마이그레이션 없음** — 코드에서 흡수.

### 1.2 구성형 `layout` v2 (고정 디자인 페이지 — 단순화)
> 고정 디자인 페이지로 확정 → **`mode`/`order`/`constraints`/`mobile` 오버라이드 불필요**. 전부 절대배치(상대단위) + 실제 텍스트. 모바일도 동일 캔버스를 폭에 맞춰 균일 scale(reflow 없음).
```ts
type Layout = {
  version: 2;
  canvas: { baseWidth: number; aspect: string }; // 좌표·글자 환산 기준
  pageBg?: string;
  blocks: Block[];
};
type Block = {
  id: string;
  type: "text" | "image";          // shape/frame 등은 후속
  z: number;                        // stacking 순서
  xPct: number; yPct: number; wPct: number; hPct: number; // 캔버스 % (상대단위·절대배치)
  rotation?: number; opacity?: number;
  content: TextContent | ImageContent;
};
// TextContent: { html, fontRel(=baseWidth 대비 비율), align, weight, lineHeight, color, bgColor, paddingPct }
// ImageContent: { src, fit, focusX, focusY, zoom, radius, overlayDarken }
```
- **`fontRel`**: 현행 `fontSizePx`(기준폭 440)를 비율로 변환 저장 → 화면 폭에 비례 scale(잡지처럼 페이지 전체가 균일 확대/축소).
- **읽기 호환**: `parsePageLayout`가 `version` 없음/1 = 구버전 → 메모리에서 v2 업그레이드(좌표 % 유지, `fontSizePx/baseWidth` → `fontRel`). **새 저장은 항상 v2.**
- **Zod**: 블록 타입별 스키마 엄격 검증(`zod/v4`). 불량 블록 필터(현행 동작 유지).

### 1.3 목차(TOC) locator 재설계 — **스키마 변경(additive, migrate deploy)**
- 현행 `MagazineTocEntry.pageNumber`(문자열 느슨 매칭) → **`pageId` FK 추가**(`onDelete: Cascade`). `blockId?`(구성형 섹션 점프용, nullable) 추가.
- 공통 locator `{ pageId, blockId? }` 로 이미지형/구성형이 **동일 목차 UI** 공유. 구성형은 heading 블록에서 자동 생성 옵션, 이미지형은 페이지+제목+썸네일 수동/자동.
- 마이그레이션: `pageId` nullable 추가 → 기존 entry는 `pageNumber`로 1회 백필 → 이후 FK 사용. (raw SQL + 체크섬 등록 절차, 공유 DB 규율 준수)

### 1.4 스키마 변경 총량 (최소)
- ✅ 변경 없음: `Magazine`, `MagazinePage`, 구성형 `layout`(Json 진화), 캔버스 메타(layout.canvas에 보관), 리더 설정(localStorage).
- 🔧 migrate deploy(additive): `MagazineTocEntry.pageId`(+`blockId?`). 죽은 `web` enum 정리(코드만, enum 값 제거는 보류 — 드리프트 주의).

## 2. 뷰어 명세 (구현 1순위)

### 2.1 공통 셸 `MagazineReaderShell`
- 외곽 1벌: 헤더(제목·닫기), 하단 진행률바(N/M·%), 컨트롤(줌·목차·페이지이동·풀스크린·다크·설정), **오버레이 토글**(모바일 탭/스크롤 자동숨김), 키보드(데스크톱 ↑↓/PgUp·Dn/←→).
- 내부 페이지 렌더 + 목차 UI만 `kind` 분기. **page.tsx 하드코딩 분기·`magazineId` 미전달 버그·데드코드(`PageSpread`/`ViewerControls`) 정리.**
- 딥링크 `?page=` 유지(startIndex 클램프). 위치 복귀 = **페이지 앵커/blockId 기반**(픽셀 오프셋 저장 금지).

> **두 종류 동일 경험.** 사용자는 구성형/이미지형을 구분하지 않음 — 하나의 "매거진 뷰어". 아래 동작은 두 종류 공통이고, 페이지 렌더만 다름(이미지형=비트맵 `<img>`, 구성형=실제 텍스트 얹은 고정 캔버스 `ComposedPage`).

### 2.2 페이지 데크 `PageDeck` (공통 페이저)
- **데스크톱**: **한 페이지 / 두 페이지 토글**(기본 두 페이지=양면 스프레드). **실물 잡지 넘김 효과**가 두 종류 공통 기본(이미지형·구성형 모두). 키보드 ←→/PgUp·Dn.
- **모바일**: **단면 페이지 스와이프**(한 장씩). react-pageflip 역넘김 우회 폐기.
- **페이저 라이브러리**: react-pageflip 유지(넘김 효과) 또는 단면/스프레드 통합 페이저(Embla 등)로 대체 — 모바일 백플립 버그 해소 위해 Stage 4 V1에서 평가. 어느 쪽이든 **두 종류·두 디바이스 동일 페이저** 사용.
- **확대**: 핀치/더블탭(모바일)·휠/더블클릭(데스크톱) **인라인 1급**. 풀스크린은 같은 컴포넌트의 모드. 구성형도 동일(작은 글자는 핀치줌으로 — reflow 안 함).
- 가이드줌(선택): focus crop 좌표 있으면 페이지 내 panel-by-panel.

### 2.3 페이지 렌더 — 종류별 차이(이것만 다름)
- **이미지형(`kind=image`)**: `imageUrl` 비트맵을 페이지 프레임에 표시.
- **구성형(`kind=composed`)**: `ComposedPage`로 **고정 디자인 캔버스를 폭에 맞춰 균일 scale**(데스크톱·모바일 동일, reflow 없음). 블록은 **실제 텍스트로 렌더**(선명·선택 가능·복사·SEO·스크린리더) → 동시에 RAG/챗봇이 읽는 데이터. **독자 글자/줄간격/여백/테마 설정 없음**(편집자 디자인 의도 보존).

### 2.4 목차·페이지이동·진행률
- 목차: **두 종류 동일한 통일 리스트**(행 = 작은 썸네일 + 제목 + 쪽/섹션번호). 이미지형은 수동 `TocEntry`(제목+쪽), 구성형은 섹션/heading에서. 데스크톱 우측 사이드패널 / 모바일 하단 시트. **pageId locator** 기반.
- 페이지이동: 좌우 클릭영역 + 하단 버튼 + 키보드 + 썸네일 점프. Kindle Page Flip식 "위치 보존 미리보기" = 썸네일 패널(에디터와 동일 컴포넌트 지향).
- 진행률·이어보기 + 호 끝→다음 호 배너(검토).

### 2.5 모바일 제스처 규약(공통 강제)
탭=오버레이 토글 · 스크롤 시작=UI 자동숨김 · 핀치=줌 · 더블탭=2x 부분줌→복귀 · 스와이프=넘김. **가시 컨트롤 항상 제공**(숨은 제스처 전용 금지), 최초 1회 힌트. 줌 중 스크롤/넘김 잠그는 **상태기계**로 제스처 충돌 방지.

### 2.6 성능
전 페이지 동시 렌더 → **가상화/지연로딩**(viewport 진입). 썸네일은 경량 별도 이미지. 줌/제스처 단일 구현으로 통합(인라인 vs 라이트박스 중복 제거).

### 2.7 뷰어 컴포넌트 구조(안)
```
MagazineReaderShell (공통 외곽·컨트롤·오버레이·키보드·진행률·줌·한/두 페이지 토글)
├─ TocPanel    (pageId locator, 두 종류 통일 리스트)
├─ ZoomController (상태기계, 핀치/휠/더블탭)
└─ PageDeck    (공통 페이저: 데스크톱 넘김·스프레드 / 모바일 단면 스와이프)
   ├─ ImagePage    (kind=image: 비트맵)
   └─ ComposedPage (kind=composed: 실제 텍스트 얹은 고정 캔버스, 균일 scale)
```

## 3. 에디터 명세 (구현 2순위, 데스크톱 우선)

### 3.1 단일 에디터 셸 `MagazineEditor`
- 페이지별 라우트(`/pages/[pageId]` push) **폐기** → 한 화면. 좌/하단 **페이지 썸네일 패널**로 라우트 전환 없이 스위칭. 편집 중 다른 페이지 항상 보임.
- 페이지 패널 2모드: 썸네일 레일(빠른 전환) / 그리드뷰(대량 재정렬·복제). dnd-kit(현행 `PageListSortable`·`ComposedPageManager` 자산 재활용). 모바일 ↑↓ 폴백(페이지 관리만, 결정 #1).
- 페이지 컨텍스트 메뉴: 빈 페이지 추가 / 복제 / 삭제 / 위·아래 / 제목. 빈페이지 vs 복제 명확 구분.

### 3.2 캔버스 편집(구성형, 데스크톱)
- 8방향 핸들 리사이즈(Shift=비율) · 드래그 이동 · **기본 스냅 가이드**(요소·캔버스 기준) · Position 패널(상대단위 직접입력+비율잠금) · z-order · 정렬.
- 라이브러리: `moveable`+`selecto` 또는 `react-rnd` — Stage 3에서 평가. 키보드 nudge·다중선택 가능하면 포함. **undo/redo·스마트가이드·Magic Resize 1차 제외.**
- 텍스트: raw HTML textarea → **인라인 WYSIWYG(Tiptap, 블로그 자산 재활용)**. 글자 `fontRel` 저장.
- 이미지: focus crop을 **프레임(클립) 개념**으로 일반화.

### 3.3 모바일 오버라이드 — **삭제됨**
- 구성형이 고정 디자인 페이지로 확정(reflow 없음)되어 **모바일 전용 편집/오버라이드가 불필요**. 에디터는 데스크톱 고정 캔버스 설계 하나만 다룬다. 그 결과가 모바일에서도 폭맞춤 scale로 그대로 보임. (결정 #3 개정)

### 3.4 저장
- 자동저장(드래프트) + 이탈 시 미저장 경고. **페이지 단위 부분저장** 우선(P2028 5s 한도, statement 최소화). 정렬은 현 2-pass `applyPageOrder` 존중.

### 3.5 이미지형 통합
- 업로드·정렬·파일명·삭제(`PageUploadZone`/`PageListSortable`) 유지 + 같은 페이지 패널 셸에 흡수("한 콘텐츠=여러 페이지" 멘탈모델 통일). 블록 에디터 강제 안 함. image→composed 전환 1차 제외(결정 #7).
- 자산 업로드 경로 통일 검토(이미지형 sharp WebP vs 구성형 blog 업로드 재사용 → 매거진 전용).

### 3.6 정리
죽은 `web` enum(코드), 데드코드, `layout.version`, Zod 검증 도입.

## 4. Stage 4 구현 Phase (각 별도 게이트)

**뷰어 먼저:**
- **V1** — 공통 셸 + **고정 페이지 페이저**(데스크톱 넘김·스프레드 / 모바일 단면 스와이프, 한·두 페이지 토글) + 인라인 줌 + 통일 목차(pageId locator) + 진행률 + 가상화 + 이미지형 페이지 렌더. (페이저 라이브러리 결정 포함)
- **V2** — **구성형 페이지 렌더**(실제 텍스트 얹은 고정 캔버스, 균일 scale)를 같은 페이저에 통합 + `layout` v2 읽기호환. (reader 설정·flow 폴백 없음)

**에디터:**
- **E1** — 단일 에디터 셸 + 페이지 썸네일 패널 + dnd 재정렬 + 이미지형 통합.
- **E2** — 구성형 캔버스 편집(핸들·스냅·Position·z·정렬) + 텍스트 WYSIWYG + 이미지 프레임 + `layout` v2 저장 + Zod + 자동저장·이탈 경고. (모바일 오버라이드 없음)

**선행 공통:** TOC `pageId` FK 마이그레이션(additive), `parsePageLayout` v2 읽기호환, 죽은 코드/enum 정리.

## 5. 리스크 / 구현 시 주의
- `layout` v2 전환: 반드시 읽기호환 먼저(구버전 무중단), 새 저장만 v2. 39호 등 기존 구성형 데이터 보존.
- 이미지형 경로는 **절대 회귀 금지**(렌더/업로드/정렬 무변경 원칙).
- 페이저 라이브러리 교체 시 딥링크·키보드·스프레드·접근성 회귀 점검.
- 공유 운영 DB: TOC 마이그레이션은 raw SQL + 체크섬 등록 절차(`migrate deploy` 직접 실행은 풀러에서 행업).
- 모바일 제스처 상태기계: 줌/스크롤/스와이프 충돌 테스트(현행 중복 구현이 충돌원).
- **접근성/고정 레이아웃(의식적 결정)**: 구성형을 고정 디자인 페이지(reflow 없음)로 확정 → 이미지형·Apple News PDF이슈와 동일한 fixed-layout 트레이드오프 수용. 완화책 = **실제 텍스트 렌더**(스크린리더·확대·복사·SEO) + 핀치줌 + (후속) 페이지 alt/캡션. WCAG 리플로우(1.4.10) 엄격충족 대신 편집자 의도 + 실제 텍스트로 절충.
